## はじめに

お仕事上、UT（単体試験）を実施する際には、テストツールのJestやJunitを使用して、テストの効率化を図ろうとする。でも、テストコードを作るにも時間がかかるためきちんと人の手で作りこもうとすると納期に間に合わない問題に直面し、結局は１項目１項目テスターがデバッグしてUTを行っていた。

最近はテストコードをＡＩに書かせる方法をよく耳にするので、フロントエンドのテストをＡＩを使ってＵＴを実施し、その過程で得た知見を書いてみる。

---

## 作業内容

### Vitest + React Testing Library によるテスト環境構築

#### なぜVitestを選んだか

Next.js（App Router）環境では、従来のJestよりも**Vitest**の方が相性が良い。設定がシンプルで、ESMネイティブ対応であることに加え、ウォッチモードでファイル保存のたびに自動でテストが再実行されるため、TDDサイクルを非常に効率よく回せる。

#### UT設計の5つの観点

コンポーネントのUTを書く際に意識した観点を整理すると以下の5つになる。

| 観点 | 内容 |
|------|------|
| 表示・非表示 | 条件によって要素が表示/非表示されるか |
| テキスト表示 | 正しいテキストが描画されているか |
| 条件分岐 | propsの値によって描画が切り替わるか |
| リンクのhref | aタグやnext/linkのhrefが正しいか |
| ユーザー操作 | クリックなどのイベントが正しく発火するか |

### ArticlePreview のバグ修正

テスト実装の中で `ArticlePreview` コンポーネントのバグも発見・修正した。テストを書くことで普段の手動確認では見落としがちな細かい不具合を炙り出せるのは、UTの大きなメリットだと感じた。

---

## 学んだこと

- **VitestはNext.js環境に最適**：Jestより設定が軽量で、ESM対応・ウォッチモードによるTDD効率化が強み
- **UT設計の5観点**：「表示・非表示 / テキスト / 条件分岐 / href / ユーザー操作」を軸にすると設計が安定する

---

## まとめ

Vitest + React Testing Libraryの組み合わせは、Next.js App Routerとの相性が良く、導入コストも低いためおすすめな方法。テストを書く文化を早い段階でプロジェクトに根付かせることで、実装変更時の安心感が大きく変わることを実感している。
