## はじめに

PC活動ログを自動収集するアプリを開発・運用しているのだが、コレクタープロセスが落ちたときに自動で再起動する仕組みが不十分だった。「Windowsサービスにすればいいのでは？」と思ったのだが、実はそこに大きな落とし穴があった。今回はその問題を解決した過程と、あわせて実施したLambdaプロンプト改善・Vercelデプロイ修正の知見をまとめる。

---

## 作業内容

### 1. コレクタープロセスの自動再起動問題をタスクスケジューラで解決

#### Windowsサービスにできない理由（Session 0 Isolation）

最初に思いついたのはWindowsサービスへの登録だった。しかしWindows Vista以降、サービスはすべて **Session 0** で動作する。Session 0はデスクトップを持たない隔離された環境であり、以下の操作が根本的に不可能になる。

- 画面キャプチャ（`BitBlt`系のAPI）
- アクティブウィンドウのタイトル取得（`GetForegroundWindow` など）
- デスクトップGUIへのアクセス全般

PC活動ログの収集にはまさにこれらが必要なため、サービス化は断念した。

#### タスクスケジューラ＋VBScriptという解法

ログインユーザーのセッション（Session 1以降）でプロセスを動かすには、**タスクスケジューラ**が有効な手段となる。「ログオン時にトリガー」＋「特定ユーザーで実行」の組み合わせにより、ログインセッション上でプロセスを起動できる。

ただし、そのまま`cmd.exe`経由でexeを起動すると黒いコンソールウィンドウが一瞬表示されてしまう。これを回避するためにVBScriptを噛ませる。

```vbscript
Dim objShell
Set objShell = CreateObject("WScript.Shell")
objShell.Run "C:\path\to\collector.exe", 0, False
Set objShell = Nothing
```

`Run`の第2引数`0`が`SW_HIDE`（ウィンドウ非表示）に対応する。`wscript.exe`でこのVBSを実行することで、コンソールウィンドウを一切表示せずバックグラウンド起動が実現できる。

#### VBScriptのエンコーディング問題

ここで1つハマりポイントがあった。VSCodeで作成したVBSファイルをデフォルト（UTF-8）のまま保存したところ、日本語コメントを含んでいたため実行時にエラーコード `80070003` が発生した。

VBScriptのランタイム（`wscript.exe` / `cscript.exe`）はファイルをデフォルトで **Shift-JIS** として読み込む。UTF-8のBOMなしファイルに日本語が含まれると文字コードが不一致となりパーサーエラーになる。

**解決策は2つ：**

1. VSCodeで「エンコード付きで保存」→「Shift JIS」を選択する
2. PowerShellで明示的にエンコーディングを指定して書き出す

```powershell
$content = @"
Dim objShell
Set objShell = CreateObject("WScript.Shell")
objShell.Run "C:\path\to\collector.exe", 0, False
"@
[System.IO.File]::WriteAllText(
    "C:\scripts\start_collector.vbs",
    $content,
    [System.Text.Encoding]::GetEncoding("shift_jis")
)
```

Shift-JISで保存し直したところ、エラーなく動作するようになった。

### 2. AWS CDKによるLambdaデプロイの流れを再確認

#### `pnpm run deploy` で自動トランスパイルされる

AWS CDKプロジェクトでは `cdk.json` の `app` キーにトランスパイルコマンドが仕込まれていることが多い。今回のプロジェクトでは `pnpm run deploy`（内部で `cdk deploy`）を実行すると、TypeScriptのコンパイルが自動で走るため、**別途 `build` コマンドを実行する必要がない**。

これを知らないと「ビルドしたのにデプロイされていない」「ビルドコマンドを二重に叩いてしまう」といった混乱が生じる。

#### コミットだけではLambdaは更新されない

GitHubにコードをpushしただけではAWS上のLambda関数は更新されない（CI/CDが構成されていない場合）。明示的に `cdk deploy` を実行して初めてAWSリソースへの変更が反映される。当たり前のことのようだが、開発中に「コード直したのに動きが変わらない」という場面で見落としがちなポイントである。

### 3. Vercelデプロイのパスを正しく把握する

#### `packages/web` から実行すると別プロジェクトに接続される

モノレポ構成のプロジェクトで作業していたのだが、`packages/web` ディレクトリ内から `vercel --prod` を実行したところ、意図しない別のVercelプロジェクトに接続されてしまった。

Vercelプロジェクトはリポジトリの **ルートディレクトリ** にリンクされているため、正しいコマンドは以下のようにリポジトリルートから実行する必要がある。

```bash
# リポジトリルートで実行
cd /path/to/repo-root
npx vercel --prod
```

#### GitHubとの自動連携が有効でない場合は手動デプロイが必須

VercelはGitHubリポジトリと連携すると `main` ブランチへのpushで自動デプロイが走る。しかし自動連携が設定されていないプロジェクトでは、上記のように手動で `vercel --prod` を叩かないと本番環境に反映されない。デプロイされていると思って動作確認したら古いコードだった、という状況を防ぐために確認が必要。

### 4. Lambda分析プロンプトの改善でレポート品質を向上

以下の3点を変更した。

| 変更項目 | 変更前 | 変更後 |
|---|---|---|
| `max_tokens` | 4096 | 8192 |
| 本文の文字数ルール | 1,000〜3,000字 | 3,000字以上・上限なし |
| プロンプトに渡す情報 | 基本ログのみ | tech-notes・AIチャットログを追加 |

`max_tokens` を倍にしたことでAIが途中で打ち切られることがなくなり、tech-notesやAIとのやり取りを入力として渡すことで、単なる作業ログの羅列ではなく「なぜその実装を選んだか」「何でハマったか」という文脈が含まれた記事が生成されるようになった。

## 学んだこと

### Windowsのセッション分離はデスクトップアプリ開発で必ず意識する

WindowsサービスはSession 0で動くためデスクトップ操作系のAPIが一切使えない。画面キャプチャやアクティブウィンドウ取得が必要なツールは、タスクスケジューラ（ログオン時・特定ユーザー指定）で起動する設計にする必要がある。

### VBScriptはShift-JISを要求する（今も）

2025年現在もVBScriptランタイムはShift-JISがデフォルト。UTF-8全盛の時代に意識しないと即ハマる。日本語を含む場合はエンコーディングを明示して保存すること。

### CDKのデプロイフローを把握しないとデバッグが混乱する

「コード変えたのに動きが変わらない」はデプロイ忘れが大半。CDKプロジェクトでは `cdk deploy` のフローとトランスパイルの自動化有無を最初に確認しておくと無駄な時間を使わずに済む。

### AIへの入力情報を増やすと出力品質が上がる

プロンプトに渡す情報量を増やすことでAIの出力品質は明確に変わる。単なる「やったこと」だけでなく「なぜそうしたか」「何でハマったか」という文脈情報を渡すことが、有益なアウトプット生成の鍵になる。
