## はじめに

先日と続き。PC監視アプリのフロントエンド開発にVitestを導入してユニットテストを実施してみた件。
その検証作業を通じて得た気づきをまとめる。

---

## 作業内容

### Vitest + React Testing Library の導入

VitestをReactプロジェクトに導入し、コンポーネントおよびユーティリティ関数のユニットテスト環境を構築した。既存のコードに対してテストを書こうとしたところ、いくつかの設計上の問題点が浮き彫りになった。

---

## 学んだこと

### 1. `export`がないとユニットテストできない

TypeScriptでは、`export`を付けていない関数はモジュール外から参照できない。当然、テストファイルからもインポートできないため、ユニットテストの対象にならない。

```typescript
// ❌ テストできない
function formatDate(date: Date): string { ... }

// ✅ テストできる
export function formatDate(date: Date): string { ... }
```

テストを書くことで、「この関数はちゃんと設計されている処理なのか？」という意識が自然と生まれるため、極力エクスポートをつけてテスト対象に組み込むべき。

---

### 2. 汎用関数はユーティリティファイルに切り出す

`formatDate` のような汎用関数がコンポーネントファイルの中に埋め込まれていると、再利用もテストもしづらい。

```
src/
  components/
    ArticlePreview.tsx   ← ここで使っていた処理だけど、切り取って
  utils/
    formatDate.ts   ← ここに切り出す
```

ユーティリティファイルに切り出すことで、**再利用性・テスト容易性・責務の分離**がまとめて改善される。

---

### 3. `new RegExp()` をレンダリングのたびに生成しない

`HighlightedText` のようなコンポーネントで、正規表現をコンポーネント内部で毎回生成しているケースがあった。

```typescript
// ❌ 毎レンダリング生成される
const regex = new RegExp(keyword, 'gi');
```

キーワードが変わらないなら、モジュールスコープに定数として切り出すか、`useMemo` でメモ化することで不要な再計算を防げる。

```typescript
// ✅ モジュールスコープに切り出す（固定値の場合）
const HIGHLIGHT_REGEX = /検索ワード/gi;
```

---

### 4. コピペコードは共通化のサイン

文字数表示UIが複数箇所にほぼ同じ形で存在していた。これは共通コンポーネント化または共有定数化のタイミングだ。

```typescript
// ✅ 共通コンポーネントに切り出す例
<CharacterCount current={value.length} max={MAX_LENGTH} />
```

同じロジックが2箇所以上に出てきたら、それはリファクタリングのシグナルと捉えるとよい。

---

## まとめ

Vitestを導入してテストを書き始めると、「テストしにくいコード＝設計に問題があるコード」であることがわかる。

- `export` を意識することで関数の公開設計が明確になる
- ユーティリティ関数の切り出しで再利用性が上がる
- コピペの発見が共通化のきっかけになる

テストを書くこと自体が、コードレビューの一形態になっていると感じた。フロントエンド開発においてもユニットテストの習慣は早めに取り入れる価値がある。
